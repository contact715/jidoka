# Еженедельный план усиления jidoka — 2026-W27

Дата запуска: 2026-06-29. Режим: только план и предложения, без внедрения кода (внедрение требует отдельного «ок» владельца).
Метод: чистый клон main → авто-снимок текущего состояния движка → параллельный поиск на GitHub по 8 доменам → адверсариальная верификация кандидатов по файлам репозиториев → дебаты «прокурор / защитник / судья» → синтез ранжированного плана.
Объём прогона: 64 агента, 570 действий инструментов, 24 кандидата проверено, 28 методов разобрано, ~18 минут работы.

Сквозной принцип недели подтверждён снова: ни один внешний фреймворк не берётся целиком (Python и демоны убьют инвариант «чистый Node без зависимостей»). Берём только идею и портируем её нативным hook'ом или скриптом. Каждое предложение проверено на то, что оно не дублирует уже внедрённое.

---

## 1. Краткий вывод недели

Движок jidoka технически зрелый: замкнутая петля самоулучшения, адверсариальный стек дебатов, kernel-песочница на Seatbelt, формальная проверка andon-halt в TLA+, спек-дисциплина уровней L0–L4. Но есть одна живая дыра в принуждении правил, которая важнее любого нового механизма памяти.

«Жёсткие» гейты на сообщение коммита печатают «Commit refused» / «Hard block active», но коммит всё равно создаётся: хук не передаёт наружу код ошибки. Реально блокирует только гейт на пуш (тесты и сборка). Это не теория, плохие коммиты проходят прямо сейчас (класс ошибок `gate-claims-block-but-passes`).

Поэтому неделя выстраивается по зависимостям:

1. Сначала закрыть баг с кодом ошибки в commit-гейтах и завершить калибровку 7 неизмеренных судей (это предусловие, явно названное в прошлом ресёрче).
2. И только потом строить три открытые рекомендации памяти.

Из 12 рекомендаций прошлого отчёта 9 уже внедрены, их заново не предлагаем. Реально открыты и прошли дебаты: семантический поиск памяти `memory-vector.mjs` (через слияние рангов Reciprocal Rank Fusion), дистилляция «провал → стратегия» reasoning-bank (с немедленным перехватом удаляемых артефактов), write-time дубль-гейт `memory-guard.mjs` (через разделение ролей автор/судья), плюс две новые архитектурные идеи этой недели: DAG-планировщик задач с критическим путём и bottom-up майнер конвенций кода.

## 2. Что усилить из сильных сторон

- **«Механизмы, а не декларации» — главный дифференциатор.** `instantiation-audit.mjs` падает на «призраках» (заявленный механизм без реального файла, сейчас 0 призраков), `meta-honesty.mjs` ловит самоподтверждающие записи в журнале. Беречь: продолжать гонять оба гейта в CI на каждой волне.
- **Адверсариальный стек лучший в классе** (3-раундовые дебаты прокурор/защитник/судья + best-of-N + reflexion-critic). Именно поэтому DeepEval/promptfoo/agent-as-a-judge были отклонены как дубли. Усилить через завершение калибровки судей, чтобы их вердикты стали измеренными, а не спящими.
- **Zero-dependency чистый Node.** CI и все гейты проходят на чистом клоне без установки пакетов. Этот инвариант убил 15 внешних кандидатов (mem0, cognee, A-mem, langgraph и др.). Защищать абсолютно: любой новый механизм — нативный порт идеи, никогда зависимость.
- **Реальная kernel-изоляция на macOS через Seatbelt** (`sandbox-run`) плюс живой хук `policy-enforce-hook` (выход с кодом 2 при записи в защищённые пути L0/security). Это доказанная модель для будущего `memory-guard.mjs` — переиспользовать тот же паттерн с кодом 2.
- **Спек-дисциплина L0–L4** с EARS-критериями приёмки как тестами, графом происхождения, закрытием волны через зелёный `verdict.json`. Усилить через DAG-планировщик, чтобы фазы спеки получили рёбра зависимостей и параллелизм по критическому пути.
- **Честная самодокументация** (DORMANT/SKIP вместо фейк-зелёного, «N из M судей измерены»). Та же философия нужна семантической памяти: проверка доступности эмбеддингов с честным откатом на лексический слой.

## 3. Слабые места по тяжести

**Критично**

1. Commit-гейты печатают «Commit refused», но коммит создаётся: хук не передаёт код ошибки. Блокирует только pre-push. Пропускает плохие коммиты прямо сейчас. Названо в ресёрче как явное предусловие для всего остального. (класс `gate-claims-block-but-passes` / `gate-block-not-enforced`)

**Высокая**

2. 7 из ~10 LLM-судей не измерены (спят). Измерены только 3 (constitutional-reviewer, debate-judge, reflexion-critic). `judge-calibration.mjs` готов, но прогон golden-кейсов не завершён. Это предусловие перед добавлением любого нового судьи и перед инъекцией дистиллированных стратегий (вердикты от некалиброванных судей токсичны).
3. Память не ищет по смыслу: внедрён только лексический слой TF-IDF (`memory-retrieve`), векторного слоя нет. Похожий урок под другим именем не всплывает до ошибки. Жёсткое предусловие: внутри границы движка нет источника эмбеддингов без ключа/демона, его надо построить первым.
4. Много ungated живых рисков без механического гейта: scope-escape (субагенты пишут в чужой worktree), piped-exit-masking (`git ... | tail` маскирует провал гейта), peer-restyle-instead-of-clone, reactive-literal-execution. Держатся только дисциплиной.

**Средняя**

5. Нет write-time дубль-гейта на запись в память (`memory-guard.mjs`). Правило «addition is not free» остаётся дисциплиной. Есть только лексический `dup-guard` по экспортам.
6. Контрастный сигнал адверсариального стека выбрасывается: проигравшие best-of-N (`dispatch-parallel-implementations.mjs:277`) и вердикты REVISE/BLOCK reflexion (`dequeue-reflexion.mjs:63`) force-удаляются. Движок платит токены за их генерацию и тут же стирает в /dev/null. После удаления траектория невосстановима.
7. Нет понятия критического пути: фазы орк-планировщика идут блоком без рёбер зависимостей между подзадачами. «Проектируй схему БД» и «пиши API» стартуют вместе, хотя одно должно предшествовать другому; независимые листья не параллелятся.
8. Нет bottom-up захвата конвенций существующего кода: вся машинерия движка top-down (спеки предписывают) или ретроспективна (уроки вспоминают). Ничего не считывает, как brownfield-репо уже пишет код. Это стоило токенов (51 скрипт в чужом стиле).

**Низкая**

9. Документы HONEST_SYSTEM_STATE (снимок 2026-05-31) и ENGINEERING_SYSTEM_ASSESSMENT (2026-05-29) отстают на месяц, написаны до внедрения 9 механизмов; счётчики агентов/скиллов/скриптов устарели. Headline-метрики «187 волн / 1080 коммитов» из боевого проекта, не из этого репо.
10. Observability засеяна, но мертва на этом репо: `compute-dora`/`compute-slos` выдают 0 / «stream absent» (телеметрия не накоплена). `_DASHBOARD.md` показывает цифры боевого проекта.

## 4. Ранжированные рекомендации

### Ранг 1 — P0 — Закрыть баг exit-кода commit-гейтов + завершить калибровку 7 судей (предусловие для всего)

- **Источник:** внутренний (jidoka-internal, precondition) — https://github.com/contact715/jidoka
- **Что делает:** Чинит передачу кода ошибки в хуках на сообщение коммита, чтобы «Commit refused» реально отклонял коммит. Защищает от piped-exit-masking (`git ... | tail` возвращает 0). Параллельно прогоняет golden-кейсы через `judge-calibration.mjs` для 7 спящих судей, чтобы их вердикты стали измеренными.
- **Киллер-фича:** Снимает живую дыру принуждения, которая пропускает плохие коммиты прямо сейчас, и разблокирует все три открытые рекомендации памяти. Без этого reasoning-bank кодифицирует токсичные вердикты, а memory-guard стоит на некалиброванном фундаменте.
- **Какой пробел закрывает:** Критический ungated класс `gate-claims-block-but-passes` + долг 7/10 спящих судей.
- **Точка встройки:** Commit-msg хук в `.husky/` (исправить передачу кода ошибки, добавить `set -o pipefail` против piped-exit-masking). `judge-calibration.mjs` (прогнать golden-кейсы по 7 судьям, опционально с logprob-взвешенным CoT-скорингом G-Eval и дебиасом позиции PriDe для парных судей дебатов/best-of-N).
- **Усилие:** среднее. **Приоритет:** P0. **Вердикт дебатов:** PRECONDITION. **Новое на этой неделе:** нет.

### Ранг 2 — P0 — memory-vector.mjs: семантический слой памяти через Reciprocal Rank Fusion

- **Источник:** rohitg00/agentmemory (идея RRF), A-mem (link-on-write как заметка-к-встройке) — https://github.com/rohitg00/agentmemory
- **Что делает:** Нативный zero-dep слой семантического поиска. За проверкой доступности заменяет `scoreItem()` в `memory-retrieve.mjs` на косинус эмбеддингов, слитый с существующим рангом TF-IDF через Reciprocal Rank Fusion (score = сумма 1/(k+rank), k=60). Ловит парафраз, синоним, другой язык, которые лексический ранг возвращает как 0, сохраняя точные попадания по классу/термину. Контракт `retrieve()`/`dedupe` не трогается (это уже задокументированная точка замены на строках 20–23). Берётся ~15 строк нативной математики, ноль строк и ноль зависимостей от agentmemory.
- **Киллер-фича:** Слияние двух рангов (лексический TF-IDF + векторный косинус), а не замена одного другим. Аддитивно и с откатом — ровно паттерн движка «честный SKIP вместо фейк-зелёного».
- **Какой пробел закрывает:** Открытая P0: похожий урок под другим именем не всплывает до ошибки. Лексический слой покрывает ~80% боли, RRF добавляет недостающие 20% семантики.
- **Точка встройки:** Новый `scripts/memory-vector.mjs` реализует замену, обещанную в `scripts/memory-retrieve.mjs:20-23`. RRF внутри `scoreItem()`. Авто-кормит 3 живых вызывающих: `hooks/session-start-digest.mjs`, инъектор UserPromptSubmit, fallback в `scripts/get-spec-context.mjs`. Жёсткое предусловие: построить источник эмбеддингов без ключа/демона внутри границы движка (model-router — это политика, а не endpoint; по всему репо 0 строк эмбеддингов) ПОСЛЕ фикса бага exit-кода и калибровки судей.
- **Усилие:** среднее. **Приоритет:** P0. **Вердикт дебатов:** ADAPT (REJECT-as-dependency: agentmemory — это полное приложение с Docker и 53-инструментным MCP, целиком брать нельзя). **Новое на этой неделе:** нет.

### Ранг 3 — P1 — reasoning-bank: дистилляция «провал → стратегия» (немедленно перехватить удаляемые артефакты)

- **Источник:** google-research/reasoning-bank (ICLR 2026, 425 звёзд) — https://github.com/google-research/reasoning-bank
- **Что делает:** Две части. ЧАСТЬ А (немедленно, дёшево, самостоятельная ценность): перехватить контрастный сигнал ДО его force-удаления — вставить сохранение артефакта (diff проигравшего worktree + сливаемый item reflexion) ПЕРЕД `git worktree remove` на `dispatch-parallel-implementations.mjs:277` и ПЕРЕД `fs.unlinkSync` на `dequeue-reflexion.mjs:63`. После удаления траектория невосстановима, поэтому «перестать удалять лузеров» ценно независимо от того, когда поедет дистилляция. ЧАСТЬ Б (после предусловия): добавить категорию `strategy` в таксономию `SECTION_RULES` в `extract-retro-memory.mjs` (~строка 47), а не новый скрипт — дистиллировать контрастные пары в стратегию с ключом по классу задачи («в классе X делай Y / избегай Z»). Инъекция стратегий гейтится за калибровкой судей.
- **Киллер-фича:** Превращает адверсариальный стек из one-shot в кумулятивный: каждый проигравший best-of-N и каждый вердикт REVISE/BLOCK становится сохранённой контрастной парой, дистиллированной в стратегию на будущее. Уникальная ценность — захват сигнала, за который движок уже платит токены, но сбрасывает в /dev/null.
- **Какой пробел закрывает:** Открытая P2 #8: проигравшие best-of-N и раунды REVISE/BLOCK выбрасываются вместо превращения в стратегии. Подтверждено grep'ом: ни один memory-скрипт не читает `docs/debates` или bestofN.
- **Точка встройки:** WRITE: сохранение артефакта ПЕРЕД удалением на `dispatch-parallel-implementations.mjs:277` и `dequeue-reflexion.mjs:63` (в `docs/memory/`). DISTILL: категория `strategy` в `SECTION_RULES` в `extract-retro-memory.mjs:~47` (переиспользование, не новый скрипт). READ: расширить top-k pull в `hooks/session-start-digest.mjs` через `memory-retrieve.retrieve()`, но инъекцию стратегий гейтить за калибровкой судей.
- **Усилие:** среднее. **Приоритет:** P1. **Вердикт дебатов:** ADAPT (порт только схемы + двух промптов контраста успех/провал, без Python/Vertex). **Новое на этой неделе:** нет (но дебаты дали критическое уточнение про немедленный перехват).

### Ранг 4 — P2 — memory-guard.mjs: write-time дубль-гейт с разделением ролей автор/судья

- **Источник:** shidingz/EDV (Execute-Distill-Verify) — https://github.com/shidingz/EDV
- **Что делает:** Скелет write-gate на запись в память по образцу `policy-enforce-hook.mjs` (доказанная модель PreToolUse, выход с кодом 2). Перехватывает запись в `.claude/memory-staging/` и в журнал: запись без вердикта (shared|private|discard) → выход 2, по умолчанию отказ. Структурное разделение ролей: автор кандидата-опыта (через существующий `extract-retro-memory.mjs`) НЕ совпадает с тем, кто впускает запись. ВЫБРОСИТЬ тяжёлую трёхролевую церемонию Execute (несколько агентов переисполняют задачу) и executor-consensus — дорого, против zero-dep духа. Дедуп оставить на УЖЕ СТОЯЩЕМ TF-IDF, НЕ на Jaccard из EDV (это шаг назад).
- **Киллер-фича:** Структурное разделение ролей НА ЗАПИСИ + дефолтный отказ: автор кандидата структурно не совпадает с тем, кто впускает запись. Этого нет ни у debate-engine (судит контент, не управляет записью), ни у memory-curator (пост-write ранжирование), ни у meta-honesty (пост-хок аудит).
- **Какой пробел закрывает:** Открытая P2 #10: нет рантайм-дубль-гейта на запись. Закрывает имя рек #10 на структурном уровне (семантика приходит с рангом 2).
- **Точка встройки:** Новый `scripts/memory-guard.mjs` по образцу `scripts/policy-enforce-hook.mjs`. Гейт вставить как обязательную фазу между `extract-retro-memory.mjs` и шагом слияния, аналогично энфорсу закрытия волны в `acceptance-verdict.mjs`. Предусловие: фикс exit-code бага + ≥2 измеренных судьи.
- **Усилие:** среднее. **Приоритет:** P2. **Вердикт дебатов:** ADAPT (выбросить тяжёлый Execute-этап и Jaccard). **Новое на этой неделе:** нет.

### Ранг 5 — P2 — DAG-планировщик задач с критическим путём (рёбра dependsOn[] в орк-планировщике)

- **Источник:** open-multi-agent/open-multi-agent — https://github.com/open-multi-agent/open-multi-agent
- **Что делает:** Портировать ТОЛЬКО идею DAG-планировщика нативно, отклонив фреймворк (TypeScript multi-LLM runtime с npm-зависимостями убил бы zero-dep инвариант). РАСШИРИТЬ `orchestration-planner.mjs` (не создавать второй оркестратор): добавить опциональный слой task-node с `dependsOn:[]` внутри build/spec фаз. Из рёбер зависимостей бесплатно выпадает: независимые листья идут параллельно (короче критический путь → ниже lead-time волны, метрика, которую `compute-dora` уже читает), зависимые ждут родителя (нет гонки «API раньше схемы»), join-точка собирает результаты перед гейтом. Добавить latency-aware вес: упорядочивать готовые узлы по длине нисходящего критического пути.
- **Киллер-фича:** Task DAG, выведенный из структуры зависимостей работы, а не из фиксированного линейного списка фаз.
- **Какой пробел закрывает:** Нет понятия зависимостей между подзадачами, топосортировки, fan-out разных подзадач, join. Подтверждено grep'ом: `dependsOn`/`topolog` только в парсерах спек-frontmatter, ни один рантайм-шедулер их не использует. Не дубль: `dispatch-parallel-implementations` — это best-of-N (N попыток ОДНОЙ задачи), `parallel-guard` — безопасность write-scope.
- **Точка встройки:** РАСШИРИТЬ `scripts/orchestration-planner.mjs`. Переиспользовать без перестройки: (a) `build-lineage-graph.mjs` для топосортировки + детекции циклов; (b) `shard-story-bundle.mjs` как слой ниже; (c) `parallel-guard.mjs` ОБЯЗАН гейтить параллельные листья, чтобы два узла не писали в один worktree (закрывает живой класс scope-escape); (d) `pipeline-contract.mjs` валидирует богатый граф E2E. Энфорс через ANTI-GHOST self-test планировщика: ацикличность, каждый узел резолвится в реального агента, join достижим из всех листьев.
- **Усилие:** среднее. **Приоритет:** P2. **Вердикт дебатов:** ADAPT. **Новое на этой неделе:** да.

### Ранг 6 — P2 — standards-mine.mjs: bottom-up майнер конвенций кода (инверсия спек-дерева)

- **Источник:** buildermethods/agent-os (4973 звезды) — https://github.com/buildermethods/agent-os
- **Что делает:** Портировать ТОЛЬКО идею bottom-up захвата конвенций, отклонив фреймворк agent-os (дублирует L0–L4 + verdict.json энфорс, тащит install-скрипты / `.agent-os/` дерево / не-pure-Node тулинг; его команды сели бы неподключёнными в 37-узловой граф = повтор истории «51 скрипт умер»). Нативный zero-dep `scripts/standards-mine.mjs` ЧИТАЕТ реальный код существующего репо и выводит его де-факто конвенции (стиль импортов, error-envelope, layout тестов, нейминг), пишет в плоские `docs/standards/*.md` + индекс, и авто-инъектирует ТОЛЬКО task-relevant подмножество в момент работы агента. Систематически энфорсит правило-сегодня-только-дисциплина «клонируй сиблинга, не переписывай» (peer-restyle-instead-of-clone).
- **Киллер-фича:** Bottom-up захват конвенций brownfield: скрипт читает реальный код репо и выводит его де-факто конвенции, затем авто-инъектирует только релевантные. Единственное недостающее направление jidoka — инверсия top-down спеков и ретроспективного recall.
- **Какой пробел закрывает:** Реальное слепое пятно для brownfield/продуктовых репо: вся машинерия движка top-down или ретроспективна, ничего не считывает, КАК репо уже пишет код. `code-map.mjs` честно говорит, что он структурный, а не про конвенции.
- **Точка встройки:** Новый `scripts/standards-mine.mjs` по паттерну read-only детерминированного аудитора. Переиспользовать: `parseModule()` из `scripts/code-map.mjs` (уже ESM-парсер) для сырого сигнала + `scoreItem()`/`buildIdf()` из `scripts/memory-retrieve.mjs` для ранжирования релевантности. Инъекция по уже живому пути: `hooks/session-start-digest.mjs` и `scripts/get-spec-context.mjs:308`. Guardrails: pure Node, плоские файлы, читать живой код как единственный источник, инъектировать только top-N. Ценность реализуется на ПРОДУКТОВЫХ инсталляциях jidoka, не на каноне.
- **Усилие:** среднее. **Приоритет:** P2. **Вердикт дебатов:** ADAPT. **Новое на этой неделе:** да.

## 6. Крупные ставки

1. **Семантическая память end-to-end** (memory-vector RRF → memory-guard → reasoning-bank distill как зависимая цепочка): построить источник эмбеддингов без ключа/демона внутри границы движка, затем RRF-слияние, затем write-time дубль-гейт на семантике, затем дистилляция контрастного сигнала. Превращает память из лексического складирования в самоулучшающийся семантический граф. Строго в порядке зависимостей и только ПОСЛЕ фикса exit-кода и калибровки судей.
2. **DAG-оркестрация с критическим путём** (рёбра `dependsOn[]` в `orchestration-planner.mjs` + latency-aware вес): перевести движок с фиксированного линейного списка фаз на task-DAG, где независимые листья параллелятся, а самая длинная цепочка зависимостей стартует первой. Снижает lead-time волны (метрика `compute-dora`) и закрывает scope-escape через parallel-guard на параллельных листьях.
3. **Завершить калибровку всех 10 судей до измеренных** (G-Eval logprob-взвешенный CoT + PriDe дебиас позиции + Bayesian per-source trust + criteria-drift итеративная рубрика): перевести адверсариальный стек с 3/10 измеренных на 10/10, что разблокирует добавление новых судей, инъекцию стратегий reasoning-bank и консенсус memory-guard. Фундамент доверия ко всему верификационному слою.

## 7. Что AI-война отклонила (дисциплина «не плоди дубль»)

Все 10 верифицированных кандидатов прошли как ADAPT (взять идею, отклонить зависимость). Ни один не взят целиком — это и есть результат дисциплины. Внутри ADAPT 4 кандидата получили явный REJECT-as-dependency:

- **langchain-ai/langgraph (Swarm handoff через Command)** — REJECT как зависимость. Механизм (`Command(goto=peer)` для прямой передачи агент-агенту) реальный, 36k звёзд, живой, но пробел узкий и тянет тяжёлый Python-рантайм. Идею прямого хэндофа можно при желании портировать позже, сам пакет — нет.
- **qualixar/superlocalmemory** — REJECT как зависимость. Bayesian per-agent trust scorer (Beta-распределение) реальный и живой, но это отдельное Python-приложение. Саму идею Bayesian per-source trust занесли в notableMethods для калибровки судей, пакет не берём.
- **Fission-AI/OpenSpec** — REJECT как зависимость. Delta-спеки с заголовками ADDED/MODIFIED/REMOVED реальны, но jidoka уже имеет L0–L4 + change-ceremony + verdict.json. Дублирует существующее спек-дерево, брать нечего кроме идеи orphan-requirement анализа (уже учтена в быстрых победах).
- **agiresearch/A-mem (как репозиторий)** — REJECT как зависимость. Link-on-write + memory evolution реальны и богаче плоского журнала, но как зависимость это лишнее. Идею links[] рёбер занесли заметкой-к-встройке внутрь рекомендации ранга 2 (memory-vector), отдельный репо не нужен.

Дополнительно agentmemory (24k звёзд) отклонён как зависимость (полное приложение с Docker и 53-инструментным MCP), взята только ~15-строчная математика RRF. autogen Magentic-One Orchestrator (outer-loop replan) — ADAPT/partial, отложен: пересекается с будущим DAG-планировщиком, отдельно тянуть Python-рантайм не нужно.
