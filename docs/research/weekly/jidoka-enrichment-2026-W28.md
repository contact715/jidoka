# Еженедельный план усиления jidoka — 2026-W28

Дата запуска: 2026-07-06. Режим: только план и предложения, без внедрения кода (внедрение требует отдельного «ок» владельца).
Метод: чистый клон main → детерминированный аудит прошлых рекомендаций → параллельный поиск на GitHub по 8 доменам (64 агента, ~4.9M токенов, 543 вызова инструментов, ~20 минут) → адверсариальная верификация кандидатов по живому коду canon-репозитория → дебаты «прокурор / защитник / судья» → синтез ранжированного плана → детерминированный разбор сессий через meta-движок → самокритика полноты.

Сквозной принцип подтверждён снова: ни один внешний фреймворк не берётся целиком. Инвариант «zero-dependency чистый Node» в одиночку отсёк package-части repomix, ruflo, OpenSpec, openai-agents-python в этом раунде и 15+ кандидатов за прошлые раунды. Берём только идею и портируем нативным скриптом (~40 строк).

## 2. Краткий вывод недели

Движок структурно здоров: eval-suite 100% (85/85 механизмов проходят к спеке), 0 «призраков» в instantiation-audit, регресс-тест commit-гейта 12/12. Вся дорожная карта памяти прошлого отчёта доехала до диска.

Но детерминированный разбор сессий даёт жёсткий сигнал, который переопределяет приоритеты: **вердикт meta-trend = REGRESSING** — гейты протекают в самом свежем периоде. По правилу движка (и по прямому указанию владельца) это принудительно поднимает **укрепление протёкших и отсутствующих гейтов ВЫШЕ любых новых фич**. Поэтому неделя строится так:

1. Сначала закрыть три дыры принуждения из разбора сессий (один протёкший гейт + два незакрытых класса).
2. Затем добить калибровку судей (долг третий отчёт подряд — он разблокирует половину замороженных кандидатов памяти).
3. И только потом — новые детерминированные порты из AI-войны (все 8 из них специально сужены до части, которая НЕ ждёт калибровки и шиппится сразу).

Хорошая новость AI-войны: из 10 разобранных кандидатов один (ACE) отсеян как уже-внедрённый дубликат, остальные 9 сужены до детерминированных нативных портов, каждый закрывает конкретную ungated дыру реальной механикой.

## 3. Что усилить из сильных сторон

- **Zero-dependency pure-Node — главный отсекающий фильтр.** В одиночку отклонил package-части всех 4 «тяжёлых» кандидатов этой недели. Держать как жёсткое правило: любой adopt — только нативный порт идеи, никогда зависимость.
- **«Механизмы, а не декларации» проверяется на самом репо.** instantiation-audit падает на любом призраке (0 живых), meta-honesty ловит самоподтверждающиеся записи. Усилить, распространив тот же exit-2 паттерн на новые гейты этой недели (guardrail-tripwire, orphan-gate detector).
- **Честная самодокументация** (DORMANT/SKIP вместо фейк-зелёного, judge-state честно показывает 0/7). Именно эта честность позволила дебатам корректно отсеять ложные «дыры». Не давать ей протухнуть — два honest-state дока устарели на 5 недель (агентов теперь 53, ~15 механизмов зашипжено после их написания).
- **Лучший-в-классе адверсариальный стек** (3 раунда прокурор/защитник/судья + best-of-N + reflexion). В этом раунде он сам дал точные verified-on-disk вердикты и отверг DeepEval/promptfoo как дубли. Усилить через завершение калибровки, чтобы вердикты стали измеренными.
- **Parallel-session safety решена на уровне фреймворка** (safe-commit + commit-lock + serial task-queue + session-lock + husky-shim). Достроить последнюю недостающую ось — механический path-gate против scope-escape.

## 4. Слабые места по тяжести

**Критично**

1. **Вердикт meta-trend = REGRESSING.** Gate coverage 60% (3/5 рецидивных классов под гейтом), regression rate 33% (1 из 3 гейтов протёк после запуска), частота инцидентов растёт (9 → 13). Правило движка: сначала укрепить механизмы, потом добавлять новые.
2. **Регресс `declaration-over-implementation`:** гейт `proof-gate.mjs` живёт с 2026-05-29, но 2026-06-06 класс рецидивировал — волной построенный гейт `selftest-reality.mjs` оказался сиротой (без ссылки в реестре, без вызова в CI). Гейт был, но не держал. Это design-defect, он важнее свежего рецидива.
3. **Калибровка судей 0/7 измерены** — стоящий P0-долг ОБОИХ прошлых отчётов. Пока <2 измерены, дистилляция стратегий приватна и нельзя безопасно добавить нового судью. Блокирует половину кандидатов этой недели.

**Высокая**

4. **`gate-bypass` (3× ungated):** запись в защищённые L0-пути через Bash-канал (`>>`, `sed -i`, `fs.writeFileSync`) в обход PreToolUse-хука, плюс обход через регистр букв (lowercase). Гейт не зарегистрирован.
5. **`scope-escape` без механического гейта:** субагенты повторно писали/коммитили в worktree параллельной сессии несмотря на явные пути. parallel-guard существует, но НЕ доказан подключённым к листам параллельного DAG.
6. **`core-property-substituted-by-scaffold` (0.98, 1д назад) ungated:** CORE_PROPERTY_GATE существует только как документ, механического enforcement «несущее свойство доставлено» нет. Тот самый класс лжи, за который владелец поймал систему на projectx.
7. **22 ungated живых риска держатся только дисциплиной.** Топ по давлению: browser-verification-skipped (1.82), peer-restyle-instead-of-clone (1.66), reactive-literal-execution (1.66).

**Средняя**

8. **`ledger-pollution` (2× ungated):** строки телеметрии маскируются под записи ошибок в meta-mistakes.jsonl без обязательных полей claimed/real/caught_by. Рецидив в тот же день.
9. **buildDag() выводит рёбра DAG из трёх хардкод-меток** (data→backend→frontend), не трогая реальный код — это буквально паттерн core-property-substituted-by-scaffold внутри самого планировщика.
10. **Открытый цикл в retrieval памяти:** get-spec-context делает фиксированный top-3 однократный fetch без петли и без учёта пробелов. Код зациклен (gate-loopback), вопросы зациклены (clarify-gate), а память тянет один раз.
11. **Развязка гейтов:** change-ceremony и spec-amendment-gate НЕ подключены к .githooks/pre-commit = мёртвые скрипты. Это тот же класс, что и регресс #2 (гейт есть, но не подключён).

**Низкая**

12. **Наблюдаемость мертва на canon-репо:** compute-dora/compute-slos дают 0 / «stream absent». lead-time метрику, которую DAG-планировщик должен уменьшать, здесь физически нельзя измерить.
13. **Установленная копия ~/.claude/jidoka не git-репо** и не содержит docs/research/**, HONEST_SYSTEM_STATE, ENGINEERING_SYSTEM_ASSESSMENT — install-профиль их выбрасывает.

## 5. Ранжированные рекомендации по ROI

Ранжирование по рычагу (impact/effort). **Вердикт meta-trend = REGRESSING принудительно поднял три гейт-фикса (G1–G3) наверх — выше новых фич.** Затем идёт калибровочный долг (R1), затем новые детерминированные порты из AI-войны (R2–R9).

### Ранг 1 — P0 — G1: детектор гейтов-сирот (закрывает регресс declaration-over-implementation + развязку гейтов)

- **Источник:** внутренний (разбор сессий, meta-trend REGRESSING). Килл-фича недели.
- **Что делает:** новый read-only детерминированный аудитор, который находит любой скрипт-гейт (`*-gate`, `*-guard`, `*-audit`, или скрипт с собственным `--self-test`) с НУЛЁМ входящих ссылок из CI (`.github/workflows/ci.yml`), хуков (`.githooks/`, `.husky/`), реестра REMEDIES и других скриптов. Это ОБРАТНАЯ ось к instantiation-audit Class 1 (тот ловит «заявлено, файла нет»; этот ловит «файл есть, но никуда не подключён»).
- **Киллер-фича:** делает КАЖДЫЙ будущий гейт само-подотчётным. Ровно этот механизм поймал бы регресс 2026-06-06 (selftest-reality.mjs — файл на диске, ноль ссылок) в момент его создания, а не постфактум. Плюс сразу ловит развязку change-ceremony/spec-amendment-gate (слабое место #11).
- **Какой пробел закрывает:** протёкший класс `declaration-over-implementation` (единственный регресс в meta-audit) на уровне механизма, а не патча.
- **Точка встройки:** новый `scripts/gate-wiring-audit.mjs` рядом с `gate-audit.mjs` (тот проверяет курируемый список; этот — авто-открытие). Подключить в CI как hard-block. Куда лягут: **jidoka (движок).**
- **Усилие:** среднее. **Приоритет:** P0. **Вердикт:** внутренний (разбор сессий). **Новое:** да.

### Ранг 2 — P0 — G2: закрыть gate-bypass (Bash-канал записи в L0)

- **Источник:** внутренний (meta-audit: gate-bypass 3× ungated).
- **Что делает:** расширить `policy-enforce-hook.mjs`, чтобы он ловил запись в защищённые L0/security-пути не только через Write/Edit, но и через Bash-канал (`>>`, `sed -i`, `tee`, `node -e fs.writeFileSync`), плюс сравнивать пути регистронезависимо (обход через lowercase был реальным red-team find).
- **Киллер-фича:** закрывает боковой канал, которым дисциплина не управляет — субагент физически не сможет переписать CONSTITUTION через `echo >>`.
- **Какой пробел закрывает:** `gate-bypass` — рецидив 3×, гейт не зарегистрирован (самый частый ungated класс безопасности в meta-audit).
- **Точка встройки:** `scripts/policy-enforce-hook.mjs` (парсинг Bash-команды из PreToolUse payload) + регистрация remedy в `meta-remedies`. Куда лягут: **jidoka (движок) + глобальный ~/.claude** (PreToolUse-хук живёт в settings.json).
- **Усилие:** среднее. **Приоритет:** P0. **Вердикт:** внутренний. **Новое:** да.

### Ранг 3 — P1 — G3: schema-гейт против ledger-pollution

- **Источник:** внутренний (meta-audit: ledger-pollution 2× ungated).
- **Что делает:** маленький детерминированный гейт, который валидирует каждую строку `meta-mistakes.jsonl`: обязательны поля `claimed`, `real`, `caught_by`; строки телеметрии (run1/run2 и т.п.) отклоняются как не-инциденты. Exit-2 при нарушении.

- **Усилие:** низкое. **Приоритет:** P1. **Вердикт:** внутренний. **Новое:** да.

### Ранг 4 — P0 — R1: добить калибровку ≥2 судей + нативный Rogan-Gladen theta+CI

- **Источник:** UW-Madison-Lee-Lab/LLM-judge-reporting — https://github.com/UW-Madison-Lee-Lab/LLM-judge-reporting
- **Что делает:** сначала САМ прогон — разметить golden-кейсы, прогнать судей, записать `docs/evals/<slug>/calibration.json` минимум для 2 судей. Затем ~40 строк нативной функции `roganGladen({p,q0,q1})→{theta,ci}` + `budgetSplit(m)` рядом с agreement()/drift() — коррекция систематического смещения судьи (формула Rogan-Gladen 1978, учебник, не зависимость).
- **Киллер-фича:** adaptive budget-split — при фиксированном бюджете разметки делит его между sensitivity и specificity так, чтобы минимизировать ширину доверительного интервала. Превращает расплывчатое «иди откалибруй 7 судей» в конкретное «размести N кейсов вот так, чтобы сжать интервал с ±18% до ±5%». Бьёт по ПРИЧИНЕ 0/7, а не по симптому.
- **Какой пробел закрывает:** стоящий P0-долг обоих прошлых отчётов. Как только measuredJudges ≥ 2 — reasoning-distill:151 авто-переключает стратегии private→shared, разблокируется весь замороженный memory-roadmap (false-consensus, семантик-соседи A-MEM/MemR3, per-item ACE).
- **Точка встройки:** функции внутрь `scripts/judge-calibration.mjs`; реальный прогон обогащает `docs/evals/<slug>/calibration.json`; judge-calibration-state.mjs сам флипает slug в measured по наличию файла (не трогать — честный маркер). Куда лягут: **jidoka (движок).**
- **Усилие:** высокое. **Приоритет:** P0. **Вердикт дебатов:** ADAPT (high confidence, всё verified on disk). **Новое:** да (конкретный механизм разблокировки старого долга).

### Ранг 5 — P0 — R2: подключить parallel-guard как hard path-gate на листах DAG (закрыть scope-escape)

- **Источник:** OpenHands Refactor SDK (rolling-integration-branch) — https://arxiv.org/abs/2606.00953
- **Что делает:** доказать и подключить `parallel-guard.mjs` как hard path-gate на каждом листе параллельного DAG — субагент физически не может записать/закоммитить в worktree другой сессии. Опционально: protocol ветка-на-агента с PR в общую катящуюся ветку.
- **Киллер-фича:** детерминистическая конвергенция параллельных листов + write вне scope блокируется exit-2 (тот же паттерн, что policy-enforce-hook), а не ловится постфактум.
- **Какой пробел закрывает:** `scope-escape` — единственная незакрытая ось parallel-safety.
- **Точка встройки:** parallel-guard.mjs как hard path-gate на write_scope листов внутри `dispatch-parallel-implementations.mjs`; per-leaf merge через safe-commit commit-lock. Куда лягут: **jidoka (движок).**
- **Усилие:** среднее. **Приоритет:** P0. **Вердикт дебатов:** ADAPT. **Новое:** да.

### Ранг 6 — P1 — R3: guardrail-tripwire.mjs — mid-run andon-halt на нарушение несущего свойства/L0 (килл-фича)

- **Источник:** openai/openai-agents-python — https://github.com/openai/openai-agents-python
- **Что делает:** нативный zero-dep скрипт по образцу policy-enforce-hook (JSON на stdin, exit(2)=tripwire=block), но проверяет СОДЕРЖИМОЕ действия против несущего свойства волны (из MASTER_SPEC + CORE_PROPERTY_GATE.md), а не только путь. Срабатывает на per-action Write/Edit и на wave-close, обрывая дрейф ДО завершения.
- **Киллер-фича:** единственный примитив, которого у jidoka нет — валидатор, обрывающий прогон в момент нарушения несущего свойства, вместо дискретного гейта в конце. Превращает CORE_PROPERTY_GATE из документа в живой механизм.
- **Какой пробел закрывает:** `core-property-substituted-by-scaffold` (0.98, 1д назад) — держится только дисциплиной. Плюс частично scope-escape mid-run.
- **Точка встройки:** новый `scripts/guardrail-tripwire.mjs`; две точки: PreToolUse Write|Edit цепочка в ~/.claude/settings.json рядом с policy-enforce-hook, и output-guardrail вызов внутри acceptance-verdict. Куда лягут: **jidoka (движок) + глобальный ~/.claude.**
- **Усилие:** среднее. **Приоритет:** P1. **Вердикт дебатов:** ADAPT (package REJECT как Python-зависимость; порт детерминистический). **Новое:** да.

### Ранг 7 — P1 — R4: cohesionPartition() — решение split/serialize из реальных import-рёбер

- **Источник:** arxiv 2606.00953 — Cohesion-Aware Task Partitioning — https://arxiv.org/abs/2606.00953

- **Киллер-фича:** замена хардкод-каркаса buildDag (data→backend→frontend) на решение из фактического code coupling — напрямую бьёт по core-property-substituted-by-scaffold ВНУТРИ самого планировщика.
- **Какой пробел закрывает:** buildDag выводит рёбра из трёх меток, не трогая код (scaffold вместо решения из фактов).
- **Точка встройки:** `orchestration-planner.mjs` строки 112-113: вставить cohesionPartition(buildDag(task), task) между buildDag() и scheduleDAG(); переиспользовать code-map.mjs. Куда лягут: **jidoka (движок).**

### Ранг 8 — P1 — R5: детерминистическая evidence-gap петля в retrieval памяти

### Ранг 10 — P2 — R7: persisted class-edges (prevInClass[]) в memory-consolidate

- **Точка встройки:** `memory-consolidate.mjs` (~строки 189-198, пересобирается каждую сессию). Куда лягут: **jidoka (движок).**

## 7. КИЛЛЕР-ФИЧИ НЕДЕЛИ

Три рычага, каждый меняет то, КАК работает движок и вся среда на нём, а не «ещё один скрипт».

**1. Детектор гейтов-сирой (G1) — само-подотчётность каждого гейта.**
Что: read-only аудитор, ловящий любой гейт-скрипт с нулём входящих ссылок из CI/хуков/реестра. Почему рычаг: сегодня движок гарантирует «заявленный механизм имеет файл» (instantiation-audit), но НЕ «файл-гейт реально подключён». Именно эта дыра дала единственный регресс в meta-audit (selftest-reality.mjs — сирота) и держит мёртвыми change-ceremony/spec-amendment-gate. Один механизм делает КАЖДЫЙ будущий гейт на движке само-проверяемым на «wired-or-dead». Где живёт: **jidoka (движок), CI hard-block.** Класс боли навсегда: `declaration-over-implementation` (регресс) + развязка гейтов. Усилие: среднее. Приоритет: P0.

**2. guardrail-tripwire (R3) — CORE_PROPERTY_GATE из документа в живой mid-run механизм.**
Что: exit-2 валидатор, обрывающий прогон в момент, когда действие нарушает несущее свойство волны или трогает L0-путь, а не в конце. Почему рычаг: атакует напрямую самый горячий ungated класс — `core-property-substituted-by-scaffold` — тот класс лжи, за который владелец поймал систему на projectx («где недетерминированная среда?»). Превращает философский документ в andon-halt. Где живёт: **jidoka (движок) + глобальный ~/.claude** (PreToolUse). Класс боли навсегда: подмена несущего свойства каркасом. Усилие: среднее. Приоритет: P1.

**3. Замкнутая ось parallel-safety (R2 + R4) — механически невозможный scope-escape.**
Что: parallel-guard как hard path-gate на листах DAG + решение split/serialize из реального code coupling. Почему рычаг: превращает parallel-safety из «решена на 80% дисциплиной» в закрытую систему — субагент физически не может записать в чужой worktree, а параллельность выводится из фактов кода, а не из трёх хардкод-меток. Где живёт: **jidoka (движок).** Класс боли навсегда: `scope-escape` (самый опасный незакрытый класс безопасности). Усилие: среднее. Приоритет: P0.

## 12. Пробелы этого прогона (самокритика процесса)

- **MCP-журнал сессий недоступен в headless-прогоне** — разбор сессий сделан по детерминированным записям движка (meta-trend/meta-audit/ledger), а не по транскриптам живых сессий.
