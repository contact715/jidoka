# Дип-рисёрч GitHub + AI-война: чем усилить движок jidoka

> Дата: 2026-06-24. Цель: найти на GitHub репозитории и методики, которые реально
> усилят движок разработки jidoka (память, бизнес-вопросы, киллер-фичи), отсеяв
> дубли и мёртвый вес. Метод: 8 параллельных потоков дип-рисёрча → адверсариальная
> верификация каждого кандидата → дебаты (прокурор/защитник/судья) по шорт-листу →
> синтез. Машинно: 75 агентов, ~5,2 млн токенов, 764 действия, ~23 минуты.
> Воронка: 47 уникальных репо → 30 кандидатов → 18 прошли проверку → 12 в дебаты →
> 10 вердиктов ADOPT/ADAPT.

## Главный вывод

Jidoka уже на уровне или выше публичного state-of-the-art почти по всем фронтам.
Подавляющее большинство «киллер-фич», которые всплыли в поиске, у нас уже есть, часто
в более сильной форме. Поэтому **ни один внешний фреймворк брать целиком не нужно** —
все победители это Python или тяжёлые демоны, которые сядут мёртвым грузом рядом с нашим
zero-dependency Node-движком. Единый паттерн всех дебатов: **портировать одну идею
нативно в уже живой hook, зависимость отклонить.**

Реальных дыр пять, и две из них — ровно те, что владелец и подозревал:

1. **Память не умеет искать по смыслу.** Поиск 100% символьный: точное имя, ярлык
   класса ошибки, затухание по времени. Движок не может ответить «где мы уже решали
   что-то ПОХОЖЕЕ на мою текущую задачу». Релевантные прошлые уроки не всплывают
   *до* ошибки, а только ловятся гейтом *после*.
2. **Этап бизнес-вопросов — единственная фаза без forcing-function.** Это проза в
   промпте агента: нет таксономии «что спросить», нет порядка, нет проверки полноты.
3. Память чистится по возрасту, а не по измеренной пользе.
4. Имплементеру не дают плоский self-contained пакет контекста на задачу.
5. Нет рантайм-детектора зацикливания агента и нет ребра «гейт упал → дебаг → гейт» в
   машине состояний (живёт только прозой).

Куда инвестировать первым: маленький **нативный слой семантического поиска** (уроки,
спеки, ретро всплывают по смыслу на старте сессии, без Milvus/Qdrant) и **нативный
движок уточняющих вопросов** (таксономия из 9 категорий, которая гейтит фазу спеки).
Оба втыкаются в hooks и skills, которые и так запускаются на каждой волне, поэтому ни
один не станет мёртвым кодом.

## Сильные стороны (на уровне SOTA или выше)

- **Адверсариальная верификация — лучшее в классе.** Дебаты в 3 раунда
  (прокурор/защитник/судья) + best-of-N + reflexion-critic превосходят то, что даёт любой
  отдельный репо. Именно поэтому DeepEval, promptfoo, agent-as-a-judge получили REJECT —
  jidoka уже делает их работу нативно.
- **Спек-дисциплина глубже рынка.** Иерархия L0–L4 с EARS, AC-as-tests, lineage-граф,
  cascade-validate, RACI и change-ceremony строго перекрывают 12 персон BMAD и
  «LLM-оценку сложности» task-master.
- **Замкнутая петля самоулучшения** (meta-audit/log/trend/decay/generalize + In-Session
  Kaizen) — настоящий дифференциатор. Большинство memory-репо только хранят, но не ловят
  рецидив сквозь гейт.
- **Закрытие волны по вердикту реально энфорсится** (acceptance-verdict перезапускает
  proof-команду каждого AC; run-state не закрывает волну без зелёного verdict.json).
- **Безопасность и изоляция сильнее публичных аналогов:** OS-kernel Seatbelt-песочница с
  доказательством на уровне ядра + живой exit-2 PreToolUse policy-hook бьют
  git-worktree-изоляцию claude-squad.
- **Zero-dependency, чистый Node, плоские файлы** — аудируемо, гейтится, без демонов и
  ключей на горячем пути. Это реальный актив, который большинство кандидатов (Milvus,
  Qdrant, Neo4j, PyTorch-стек) бы разрушили.
- **Честная самодокументация** (движок сам пишет «N из M судей измерены, остальные
  DORMANT» и называет свои слабости).

## Рекомендации (ранжированы)

Все вердикты — **ADAPT** (портировать идею нативно, зависимость отклонить).

### P0 — быстрые победы, закрывают приоритеты владельца

**#1. Нативный слой семантического поиска** (идея из `zilliztech/claude-context` +
`ruvnet/ruflo`, без vector DB)
Маленький `scripts/memory-vector.mjs`: эмбеддит `docs/specs/**`, `meta-mistakes.jsonl`,
ретро и символы code-map; отдаёт ближайших соседей по смыслу. Brute-force косинус
(корпус — сотни доков, не тысячи, поэтому HNSW/Milvus/Qdrant — оверинжиниринг),
эмбеддинги через существующий model-router.
Киллер-фича: ответ на «где мы уже строили что-то похожее, под любым именем», который
поиск по точному имени принципиально не может.
Встройка: чтение — `hooks/session-start-digest.mjs` (каждый SessionStart) + инъектор на
UserPromptSubmit по тексту задачи; семантический fallback внутри `get-spec-context.mjs`
когда имя `--feature` неизвестно. Индекс строится по волне существующими
`extract-retro-memory.mjs` / `sync-specs-to-memory.mjs`. surface-cartographer использует
семантику как первый проход, Grep/Read — детерминированное подтверждение, чтобы
устаревший индекс не прошёл молча.

**#2. Нативный движок уточняющих вопросов** (алгоритм из `github/spec-kit /clarify`, без
установки CLI)
`scripts/clarify-engine.mjs`: таксономия из 9 категорий (каждая — Clear/Partial/Missing),
порядок вопросов по «impact × uncertainty», цикл по одному вопросу с лимитом N, формат
multiple-choice (под нетехнического владельца), атомарный write-back в версионную секцию
`## Clarifications` + машиночитаемый coverage-JSON.
Киллер-фича: машиночитаемая полнота входа — превращает «поговори с пользователем» из
добровольного поведения агента в проверяемое СОСТОЯНИЕ со стоп-критерием, который видят
гейт и meta-ledger.
Встройка: шаг 0.5 в `skills/dev-pipeline/SKILL.md` между North Star и master-spec; роли
CPO/business-process-architect/user-researcher. Плюс `clarify-gate.mjs` по образцу
`spec-first-gate.mjs`: на staged-изменения `docs/specs/**` требовать свежий
`clarify-runs.jsonl` где каждая категория Resolved или явно Deferred-с-причиной (WARN по
умолчанию, exit 1 в `--block`).

**#3. Банк вопросов + правило Mom Test для продуктовых агентов** (`deanpeters/Product-Manager-Skills`)
Смонтировать курируемый банк вопросов (Mom Test, Kano, RICE/ICE,
Opportunity-Solution-Tree, Amazon Working Backwards) как Read-and-cited референс для
продуктовых агентов; вшить правило Mom Test (спрашивать про прошлое конкретное поведение,
не про гипотезы) в промпт user-researcher. Опционально — числовая рубрика
«стоит ли решать проблему» (Frequency × Intensity × Workarounds × WTP с порогом убийства).
Встройка: оркестратор инъектит текст банка в промпты CPO/user-researcher по уже
доказанному паттерну last30days signal-brief (субагенты сами Skill вызвать не могут).
Числовая рубрика — поля в `docs/research/insights-register.json` + 5-й инвариант в
`validate-research.mjs`.

### P1 — крупные ставки

**#4. Курация памяти по измеренной пользе** (`ace-agent/ace` — алгоритм, не Python-пакет)
Поля helpful/harmful в строках `meta-mistakes.jsonl` и в библиотеке навыков; новый
`scripts/ace-curator.mjs` читает уже лежащие на диске события (agent-trace PASS/FAIL,
per-AC exitCode из verdict.json) и инкрементит счётчики; формула score меняется на
`recencyWeight × utilityFactor` со сглаживанием Лапласа; `memory-consolidate` —
delta-merge вместо полной пересборки.
Киллер-фича: память чистится по пользе, а не по возрасту (ценный старый урок больше не
затухает в DORMANT просто от времени).

**#5. Плоский story-bundle для имплементера** (идея из `BMAD-METHOD`, без кода BMAD)
`scripts/shard-story-bundle.mjs` пишет `docs/specs/stories/<wave>-<task>.story.md`,
вшивая контент get-spec-context (инлайн, не указатели) + EARS AC из ac-verify-map + строку
RACI. Это становится ТЕМ артефактом, который получает имплементер (поиск ноль раз вместо N).
Встройка: `dispatch-parallel-implementations.mjs` (~строка 172) копирует bundle вместо
сырого master-spec; Spec-First Read Gate требует, что bundle сгенерирован до правки кода.

### P2 — точечные механизмы

**#6. Рантайм-детектор зацикливания** (`OpenHands/software-agent-sdk` — только эвристики)
`scripts/stuck-detector.mjs`: 5 паттернов (повтор action+observation, action+повтор ошибки,
монолог агента, пинг-понг, цикл переполнения контекста) по скользящему окну последних
~20 вызовов. Кормится из `hooks/session-state.mjs` на PreToolUse; на порог — запись в
session-pattern-log и/или exit 42 в andon-halt. Сегодня runaway ловит только бюджетный
потолок (поздно, без диагноза).

**#7. Условный loop-back по вердикту в машине состояний** (одна идея из `langgraph`, нативно)
Изменение ~20 строк: на падение гейта роутить run-state в фазу debug и перевходить гейт
заново, а не двигаться линейно; энфорсить лимит 5 раундов, который сегодня живёт прозой.
Встройка: `orchestration-planner.mjs plan()` (дескриптор рёбер) + `run-state.mjs`
(смотреть verdict + счётчик раундов). Остальное из LangGraph (checkpoint/resume/interrupt)
у нас уже есть.

**#8. Дистилляция «провал → стратегия»** (`google-research/reasoning-bank` — два промпта)
Промпт дистилляции неудачных/исправленных траекторий в forward-looking стратегии («в
задачах класса X сначала делай Y»); скармливать раунды REVISE/BLOCK reflexion-critic и
проигравших best-of-N (сегодня выбрасываются) как контраст-пары.
Встройка: расширить `extract-retro-memory.mjs` категорией strategy; `--task` флаг на
поиске тянет top-k стратегий в get-spec-context. Зависит от #1 (семантический слой).

**#9. Top-down поиск памяти без избыточности** (`HU-xiaobai/xMemory` — принцип)
`scripts/memory-retrieve.mjs` (сначала лексическое пересечение, не эмбеддинги): фильтрует
дайджест по релевантности задаче, отдаёт по одному представителю на тему, не выбрасывая
связанные предпосылки. Метод внедрить ДО любого vector-слоя.

**#10. Гейт семантического дубля на запись** (`Durafen/Claude-code-memory` — только скелет hook)
`scripts/memory-guard.mjs` PreToolUse: на Write/Edit проверяет корпус (символы code-map +
ретро/спеки + meta-mistakes) и блокирует правки, воссоздающие существующую функцию.
Детерминированный отпечаток ПЕРВЫМ, LLM-судья опционально и только за калибровочными
гейтами. Зависит от #1 и от калибровки судей.

**#11. Soft-гейт контекст-бюджета** (`humanlayer/advanced-context-engineering-for-coding-agents`)
Цель утилизации 40–60% в промпт оркестратора + soft-гейт, предупреждающий когда фаза
прошла без артефакта компакции. Расширить run-state, не плодить второй источник правды.

**#12. Спека-как-архивная-дельта** (`Fission-AI/OpenSpec` — расширить наш change-ceremony)
`change-ceremony.mjs` (сейчас только классификатор S/M/L) дополнительно эмитит
`docs/specs/_changes/<id>/{proposal,design,tasks}.md` для M/L и архивирует на закрытии
волны — диффуемая история ПОЧЕМУ менялась спека.

## Предложение по памяти (приоритет владельца, детально)

Текущее состояние: recall 100% символьный. `memory-consolidate.mjs` ранжирует уроки только
по recency-weighted частоте (`score = Σ 0.5^(ageDays/30)`), без сравнения текста/векторов.
`get-spec-context.mjs` ищет спеки по ТОЧНОМУ имени `--feature` и идёт по цепочке parents[].
grep подтверждает: НЕТ ни embedding, ни vector, ни cosine, ни RAG.

Апгрейд (два нативных дополнения, оба zero-dependency, без демона и платного ключа на
горячем пути):
1. `scripts/memory-vector.mjs` — эмбеддинг корпуса через model-router, плоский локальный
   индекс (sqlite-vec или in-memory файл), brute-force косинус. Семантика ПРЕДЛАГАЕТ,
   Grep/Read ПОДТВЕРЖДАЕТ. Плюс шаг task-relevant поиска (сначала лексика, потом
   эмбеддинги) с параметром `--task`, top-down (по одному на тему), без выбрасывания
   связанных предпосылок.
2. Счётчики helpful/harmful на элемент + смена формулы на `recencyWeight × utilityFactor`,
   delta-merge вместо полной пересборки.
Полный vector-слой можно отложить за порог размера корпуса (~150+ строк); счётчики и
лексический task-поиск можно отгрузить сразу.
Метрика успеха: частота рецидива на класс ошибки падает, когда релевантные уроки всплывают
ДО ошибки, а не ловятся гейтом после (меряется по meta-mistakes.jsonl).

## Предложение по бизнес-вопросам (приоритет владельца, детально)

Текущее состояние: этап elicitation — единственная фаза без forcing-function. Шаг 1
dev-pipeline это проза. grep по всей установке даёт НОЛЬ совпадений на любую таксономию
вопросов или протокол интервью. PFCA — гейт готовности кода, не движок вопросов; pre-mortem
бежит ПОСЛЕ спеки.

Апгрейд (нативный движок из двух частей):
- Часть A («как/полнота»): портировать алгоритм spec-kit /clarify в
  `scripts/clarify-engine.mjs` — 9 категорий Clear/Partial/Missing, порядок по
  «impact × uncertainty», цикл по одному вопросу с лимитом, multiple-choice под
  нетехнического владельца, write-back в `## Clarifications` + coverage-JSON + трейс в
  `docs/audits/clarify-runs.jsonl`.
- Часть B («что спросить»): банк вопросов (Mom Test, Kano, RICE/ICE, OST, Working
  Backwards) как референс + правило Mom Test в user-researcher; опционально числовая
  рубрика проблемы.
Встройка: `clarify-engine.mjs` как шаг 0.5 dev-pipeline; `clarify-gate.mjs` по образцу
spec-first-gate (WARN по умолчанию, exit 1 в `--block`) — это и делает движок не-DORMANT,
потому что его зовёт pre-commit.

## Предусловие (не опционально)

**Программа калибровки судей.** Откалибровать 7 неизмеренных LLM-судей golden-кейсами
(загрузить набор DevAI 55-task/365-requirement как JSONL в agent-eval-dashboard, принять
G-Eval-style CoT-скоринг) и починить баг exit-кода «claim block but pass». Это должно
предшествовать добавлению любого нового LLM-судьи (#10 гейт дублей, per-AC судья), иначе
новый судья воспроизведёт существующий режим отказа.

## Предлагаемый порядок внедрения

1. Сначала фикс бага exit-кода гейтов + калибровка судей (предусловие, дёшево, чинит
   рецидивный промах).
